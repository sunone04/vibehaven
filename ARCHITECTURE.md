# ARCHITECTURE.md

> 系统架构顶层地图。提供域与包分层的约束规则，技术无关但结构可预测。
> Agent 在此获取"什么能依赖什么"的确定性答案。

## 设计哲学

本架构围绕一个原则构建：**通过强制执行不变量，而非微观管理实现过程，令智能体快速交付且不削弱基础。**

- 约束边界，在边界内允许实现自由
- 依赖方向单向且可机械验证
- 在边界处解析数据形状，内部信任已解析类型
- 选择可完全内化于仓库中推理的依赖与抽象

## 系统分层

每个业务域内，代码只能"向前"（向下）依赖一组固定的层。依赖方向严格单向，禁止逆向。

```
┌─────────────────────────────────────────────┐
│  UI          表现层：组件、页面、路由          │
├─────────────────────────────────────────────┤
│  Runtime     应用运行时：启动、中间件、编排     │
├─────────────────────────────────────────────┤
│  Service     业务逻辑：领域服务、用例编排       │
├─────────────────────────────────────────────┤
│  Repo        数据访问：仓储、数据源、持久化边界  │
├─────────────────────────────────────────────┤
│  Config      配置：环境变量、常量、特性开关值   │
├─────────────────────────────────────────────┤
│  Types       类型定义：领域模型、数据形状       │
└─────────────────────────────────────────────┘
        ▲ 依赖方向：上层可依赖下层，反之禁止
```

### 各层职责

| 层 | 职责 | 禁止 | 可依赖 |
|---|---|---|---|
| **Types** | 类型定义、领域模型、数据形状、枚举 | 任何业务逻辑、I/O、框架耦合 | 无（零依赖基石） |
| **Config** | 环境配置、常量、特性开关的值定义 | 业务逻辑、数据访问 | Types |
| **Repo** | 数据访问层：仓储接口、数据源适配、持久化边界 | 直接被 UI 引用、业务规则 | Types, Config |
| **Service** | 业务逻辑、领域服务、用例编排、事务边界 | 直接 I/O（通过 Repo）、直接 UI 渲染 | Types, Config, Repo |
| **Runtime** | 应用运行时：服务启动、中间件、依赖注入、编排 | 业务规则实现 | Service（及以下） |
| **UI** | 表现层：组件、页面、路由、交互状态 | 业务逻辑、直接数据访问 | Runtime, Service |

### 横切关注点：Providers

认证、遥测、功能开关、外部连接器等横切关注点，**只能通过唯一的 Providers 接口**进入业务层。

```
                    ┌──────────────┐
   认证 ──────────►  │              │
   遥测 ──────────►  │  Providers   │ ──► 注入到 Service / Runtime
   功能开关 ──────►  │  (唯一接口)   │
   外部连接器 ────►  │              │
                    └──────────────┘
```

- Providers 定义在 Runtime 层，通过依赖注入向下传递
- 业务层（Service）只依赖 Providers 的接口（定义在 Types 层），不依赖具体实现
- 禁止在业务层直接 `import` 任何横切关注点的具体实现

## 数据流向

请求自上而下穿过各层，数据形状在边界处被解析：

```
[外部请求] 
   │
   ▼
 UI ── 触发交互/路由
   │
   ▼
 Runtime ── 中间件链、依赖注入、编排
   │
   ▼
 Service ── 业务逻辑、用例编排、事务边界
   │  ▲
   │  │ Providers 横切注入（认证/遥测/开关）
   │  │
   ▼
 Repo ── 数据访问、持久化边界
   │
   ▼
[数据源] 数据库 / 外部 API / 缓存
```

**关键规则：**
- 数据进入 Service 层前，必须已被解析为已知类型（parse, don't validate）
- Service 层返回的数据应是已定型的领域对象，UI 层无需再次校验
- 错误在产生处包装为领域错误类型，向上传播时保持类型信息

## 模块间通信规则

1. **同层不直接互调**：同层模块通过 Service 层编排，不横向直接依赖
2. **跨层只向下**：上层调用下层暴露的接口，下层不感知上层
3. **接口定义在 Types 层**：跨层依赖的是接口契约，而非具体实现
4. **事件/消息**：如需异步解耦，事件类型定义在 Types 层，发布在 Service 层，订阅在 Runtime 层

## 全局约束（不变量）

以下约束须编码为 linter / 结构测试机械强制执行：

1. **依赖方向**：禁止逆向依赖，禁止跨层跳跃（如 UI 直接引用 Repo）
2. **边界解析**：系统入口处必须有数据形状解析，禁止"裸数据"进入 Service
3. **Providers 唯一入口**：横切关注点的具体实现不得被业务层直接 import
4. **命名约定**：类型、Schema、文件命名遵循统一约定（登记于下方）
5. **文件大小限制**：单文件不超过 _N_ 行（按项目设定），超出须拆分
6. **结构化日志**：日志须结构化输出，禁止裸字符串日志
